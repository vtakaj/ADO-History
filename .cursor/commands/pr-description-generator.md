# PR 説明生成ルール

このプロジェクトでは、Diff の入力に基づいて詳細で明確なプルリクエスト(PR)の説明を生成します。

## 生成方針

- 変更箇所についての説明を追記
- 変更の目的は不要で事実のみ記載
- 影響を受ける範囲について徹底的な説明を提供
- 明確さと専門性を保証
- 具体的なソースコードの行を説明に含めない
- より抽象度が高い説明に焦点を当てる
- 簡潔でありながら情報量が多い説明
- レビュアーと効果的にコミュニケーションするために必要な詳細を把握

## PR タイトル生成ルール

### 基本方針

- **コミットメッセージとは異なる**: コミットメッセージの単純な転記は禁止
- **日本語で記述**: ユーザーが理解しやすい日本語を使用
- **変更内容を要約**: 実際に変更された機能や改善点を表現
- **簡潔で明確**: 50 文字以内で変更の本質を表現

### タイトル生成の考慮事項

- **機能追加**: 「○○ 機能を追加」ではなく「○○ 機能の実装」
- **修正・改善**: 「○○ を修正」ではなく「○○ の改善」または「○○ の不具合修正」
- **リファクタリング**: 「○○ をリファクタリング」ではなく「○○ の構造改善」
- **設定変更**: 「○○ 設定を変更」ではなく「○○ 設定の更新」
- **新規作成**: 「○○ を作成」ではなく「○○ の新規実装」

### 避けるべき表現

- コミットメッセージの直接転記
- 技術的な詳細すぎる表現
- 曖昧な表現（「修正」「更新」等の単独使用）
- 英語と日本語の混在

## 出力フォーマット

```markdown
# [変更内容を要約した日本語タイトル]

## PR の概要

## 変更点

- xxx
- xxx
- xxx
```

## タイトル生成例

### 良い例

- 「ユーザー認証機能の実装」
- 「画像アップロード機能の改善」
- 「API エラーハンドリングの強化」
- 「コンポーネント構造の最適化」
- 「設定ファイルの整理」

### 悪い例

- 「feat: add user authentication」 (コミットメッセージの転記)
- 「修正」 (曖昧すぎる)
- 「update config」 (英語と日本語の混在)
- 「fix bug」 (技術的すぎる)

## 出力先

- 生成した PR 説明は `tmp/PR_DESCRIPTION.md` に保存する
- `tmp/` フォルダは git 管理から除外されているため、一時ファイルとして安全に使用できる

## 注意事項

- 変更点には事実のみ含める
- 「これにより~が簡潔になりました」のような意見は不要
- どのような変更が加えられたかを述べるにとどめる
- 日本語の口調はフォーマルでプロフェッショナル
- 出力はコードとしてマークダウンで出力
- 生成したファイルは必ず `tmp/PR_DESCRIPTION.md` に保存する
